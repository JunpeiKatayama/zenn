### 1. 低気圧お知らせ BOT

![atmospheric-pressure-man](/images/my-failed-solo-projects-2025/2025-12-19-20-32-54.png)

- 技術要素
  - コーディング: Cursor
  - 言語: Python
  - API: OpenWeatherMap
  - BOT: Discord
  - 敗因: 「低気圧」の判定ロジックを定義できず頓挫

コーディングエージェントがどうやらすごいらしい、ということで Cursor を契約して最初に作ったプロジェクトです。
体調が悪いので気圧が低いはずだ、と逐一連絡してくる友達がおり、実際どうなのかというのがすぐにわかるといいな、と考えて作りました。
気象情報を扱う API は無数にあるため、これは絶対に作れるだろうという自信とともにスタートしました。

・・・ところで皆さん「気圧が低い」とは具体的にどういう状態を指すかご存知ですか？

> 低気圧とは、周囲よりも気圧が低く、閉じた等圧線で囲まれたところをいいます。そのため、気圧が ○○hPa 以下であれば低気圧というわけではありません。
> 周囲よりも気圧が低いものの、等圧線が閉じておらず、中心が特定できないようなところは低圧部といいます。
> https://zutool.jp/column/glossary/low-pressure_high-pressure より

私は単に「xxhPa 以下であれば低気圧！」と呼べると思っていました。
なので API から現地の気圧情報を取れば良い、と考えていたのですが、どうやらここで言う「低気圧」とは相対的なものであり、周囲の気圧との差分を見るなどの専門的な知識が必要だったようです。

ということでこれ以上頑張る気になれず、頓挫しました。

### 2. 川柳投稿 WEB アプリ「センリュウキング」

![senryu_king](/images/my-failed-solo-projects-2025/2025-12-19-20-41-43.png)

- 技術要素
  - コーディング: Cursor
  - フロント: Next.js + TypeScript
  - バックエンド: Next.js API Routes
  - 認証: Firebase Auth
  - DB: Firestore
  - ストレージ: Cloud Storage
  - その他: Genkit(Gemini)
  - デプロイ: App Hosting(Firebase の Next.js 向けサービス)
  - 敗因: ユーザが驚くほど増えずサービス継続の意味消失

前述の気象予報 BOT の作成時に僕と Cursor の絆は誰にも断ち切れないレベルにまで深まっていました。
僕の創作意欲は掻き立てられ、根拠のない自信だけを根拠にして川柳投稿 SNS を作ろう！と息巻いて作業していました。

意外なほどあっさりとアプリケーションは完成しました。
公式の X アカウントも作って毎日投稿していたのですが、結局ユーザ数は 19 人で頭打ちとなりました。
そのうち 15 人程度は僕の身内で、目も当てられないような投稿ばかりになり、3 ヶ月程度でやる気を喪失してサ終しました。

https://x.com/senryu_king/status/1903762635683942908?s=20
アカウントは思い出として残しており、哀愁漂っております・・・。

サーバの維持費に月 1,500 円程度かかっており、このまま意味のないプロジェクトを続けるのもなあ・・・という気持ちになったのも大きかったです。

宣伝にコストを払わなかったりというのも敗因のひとつとは思いますが、そもそも「川柳を投稿できる SNS です！」って実はそれ X でもできるじゃんっていう話で、プロダクトの個性に欠けていたのが最も大きな敗因かなと思います。（もちろん他にも色々と荒い部分はありましたが）

ただ、やっててよかったなと思うこともありました。

「センリュウキング」のロゴは『コブラフォント』というフォントをほぼそのまま利用していました。
https://webayan39.com/font/20/

『コブラフォント』は、利用に際して、制作者(EBAYAN さん)の X アカウントのフォローが必須となっています。
なのでフォローさせていただいたのですが、なんと「センリュウキング」をかなり熱心に使っていただけたんですよね。

それだけではなく、サービス終了のお知らせをした際も励ましの連絡をくださったりと、優しさにとっても感動しました！！普通にちょっと泣きました。
こちらはフォントを使わせていただいている身なのにとてもありがたかったです。
本当にお世話になりました・・・！

全員フォローしてください 🙇
https://x.com/_EBAYAN_

### 3. Discord から操作できる Minecraft サーバー

![minecraft-gcp-server](/images/my-failed-solo-projects-2025/2025-12-19-21-10-45.png)

- 技術要素
  - コーディング: Cursor
  - 言語: Python
  - デプロイ: GCP
    - Functions
    - Scheduler
    - Storage など
  - その他: Terraform
  - 敗因: Realms(公式のマルチプレイ用サーバ)安すぎ

Minecraft のマルチプレイ用のサーバを自分で管理しよう！と思い立って初めたプロジェクト。
GCE に建てることもあり、起動時間を最適化したい。
ということで、Discord の BOT 経由でサーバの起動・終了・バックアップや、起動後の IP アドレスの取得ができるようにしました。

…が、マイクラをまともに動かすためにインスタンスサイズを上げ続けた結果、 3,000 円/月 ほどのインスタンスを契約せねばならぬこととなりました。

Minecraft 公式のマルチプレイ用サーバ Realms は 1,000 円/月 で借りられる上、24/365 で起動しているしスペックもマイクラに最適化されていて必要十分です。

あ～だからみんな Realms or 自宅サーバ なんだなあと理解して撤退しました・・・。

### 5. AI 小説連載 WEB アプリ

![sail](/images/my-failed-solo-projects-2025/2025-12-19-21-30-18.png)

- 技術要素
  - コーディング: ClaudeCode
  - フロント: Next.js + TypeScript
  - バックエンド: Next.js API Routes
  - DB: Supabase
  - デプロイ: Vercel
  - 敗因: AI の書く小説が面白くない

プロンプトエンジニアリング的な言葉を X でちらっと見た僕は、なんだか自分ならうまいことプロンプトを調整して、ユーザの反応によってその先の展開が変わるような AI の連載する小説アプリを作れるのではないか、と、また根拠のない自信を持って開発しました。

作家のペルソナを定義して、小説を書かせるとともに次回に載せるための「前回のあらすじ」を DB に持ち、2 話目以降はそのあらすじとコメントを元にした小説を生成させる・・・みたいな思想で作っていたと思います。

AI の創作作文能力が低いことは今では結構知られていますが、僕は知らなかったんですねえ・・・。

### 6. チャットツールの通知フィードアプリ

![one-chat](/images/my-failed-solo-projects-2025/2025-12-19-21-36-56.png)

- 技術要素
  - 言語: Typescript
  - FW: Electron
  - 敗因: 当然に、組織に許可を貰わないとつかえない

こちらはごく最近のもの。
友人が複数の組織で仕事をしており、Teams と Slack どちらも気にせねばならず大変である、とぼやいていたので試しに作ってみたアプリ。

![one-chat-image](/images/my-failed-solo-projects-2025/2025-12-19-21-40-14.png)
(記載されているのはテストデータです)

Teams と Slack から API 経由でメッセージを取得してフィードを作成し、クリックするとハイパーリンクで対象のアプリのメッセージに飛ぶ、というものを想定していました。
しかし、Slack ではメッセージの通知を取得するためには組織の許可が必要であり、すべての組織に事情を説明したうえで API キーをもらうというのはどう考えても不可能だろうということで頓挫しました。

こんなことよく考えれば当然の制約なのですが、なぜやる前にきづけないのか・・・。
